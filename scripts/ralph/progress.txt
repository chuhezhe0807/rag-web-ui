## Codebase Patterns
- nacos-config/ 整目录在 .gitignore（第 64 行）中被忽略；配置以 Nacos 控制台为真实源，仓库里的 yaml 仅作参考/同步用，不要期望 git 追踪
- rag-ai-service 用 uv 管理依赖（见 pyproject.toml / uv.lock），跑脚本用 `uv run python ...`；requirements.txt 保留作为 Docker 兼容，由 `uv export` 生成
- app/core/config.py 的 env_file 用 `_SERVICE_ROOT / ".env"` 绝对路径，避免 uvicorn 或测试在任意 cwd 下读不到 .env
- Settings 字段优先从 os.getenv(...) 取，缺省值作为 pydantic-settings 的 fallback；给字段改默认值时注意两层都要改
- 本地 .env 是 Settings 的真正来源；nacos-config/*.yaml 是运行时热更新快照，敏感信息（OPENAI_API_KEY 等）只进 .env，不进 Nacos
- Java 侧敏感配置用 Spring 占位符 `${ENV_VAR:}` 从环境变量注入，nacos-config 里不写明文；bean 创建处再做 `StringUtils.hasText` 校验 + fail-fast
- Maven 构建顺序：`rag-service`(parent pom) → rag-common → rag-common-service → 业务模块；缺 parent pom 时先 `./mvnw -N -DskipTests install -f pom.xml` 装根 pom
- `-DskipTests` 只跳过执行不跳过编译；要彻底跳过测试编译用 `-Dmaven.test.skip=true`。业务模块必须显式依赖 spring-boot-starter-test(test scope)，否则 src/test 骨架类连编译都过不了
- ragwebui schema 真实源：.dockervolumes/mysql/sql/01_schema.sql（由 alembic head 编译而成）+ alembic 自身；二者须同步演进。01_schema.sql 末尾 `alembic_version` 种入当前 head，保证容器首启后 `alembic upgrade head` 是 no-op
- docker-entrypoint-initdb.d 下的 SQL 只在 mysql data 目录为空时触发；已经有数据的环境改 schema 需手动 `docker exec -i <db> mysql -uroot -proot < file.sql`（CREATE TABLE IF NOT EXISTS 可保证幂等）
- Feign 客户端路径 = 目标 controller 的 `@RequestMapping` + 方法 `@GetMapping/@PostMapping`；只写方法级路径会被网关当成根路径匹配→直连服务时也对不上。审 Feign 路径务必对照目标 controller 的类级 RequestMapping
- 职责分工：KB 的 CRUD / upload / preview / process 归 Java（rag-knowledge-service + rag-document-service），Python 侧只做 chat / retrieval / api-keys / openapi。碰到 KB 写操作代码在 Python 里出现 → 基本都是迁移遗留，该删
- rag-ai-service/app/main.py 的总 prefix 是 settings.API_V1_STR (= /api/ai)，再叠加 api_v1/api.py 里各 router 的 prefix；看 Python 端点的完整路径要两段拼起来算
- schema 三方对齐的真相来源：Python ORM (`rag-ai-service/app/models/*.py`) / Java @TableField (`rag-*-service/.../entity/*.java`) / MySQL 01_schema.sql + alembic；改列的正确姿势是三处同步，并跑 scripts/ralph/check_schema.sql 与 expected_schema.md 比对
- 新增 alembic migration 的三件套：(1) 写 .py 文件并 set revision / down_revision；(2) 同步更新 01_schema.sql（DDL + 末尾 alembic_version seed）；(3) 对已存在的 DB 跑 ALTER + UPDATE alembic_version，否则 `alembic upgrade head` 会把刚做过的 DDL 再来一次
- JWT 路径分工：网关只做"存在性校验 + 原样透传"（JwtAuthGlobalFilter，order=-100，白名单 /api/ai/auth/**），业务服务 AuthInterceptor 才做"解析 + 查用户"。别在网关里做完整校验——网关不该依赖 UserServiceClient，两处做会产生两个真相源
- rag-gateway 是 reactive(WebFlux)，**不要**引 rag-common-service（Spring MVC）会 auto-config 冲突；只引 rag-common 拿 Result / GConstants / ErrorConstants 即可
- 服务间 Feign 透传 header 用 `com.chuhezhe.ragcommonservice.config.FeignConfiguration#authHeaderRelayRequestInterceptor`，从 `RequestContextHolder` 取 ServletRequestAttributes，null 时（非 web 线程）静默跳过

- **Learnings for future iterations:**
  - Gateway 是 reactive(WebFlux)、业务服务是 servlet(Spring MVC)，两边别
    互相引对方的 starter：gateway 引 rag-common-service 会触发 Spring
    Boot 对 DispatcherServlet 的 auto-config，跟 WebFlux 起冲突
  - Feign RequestInterceptor 里取 ServletRequestAttributes 一定要判 null，
    否则定时任务 / MQ 触发的 Feign 会 NPE
  - 在 WebFlux GlobalFilter 里写 response body：用 `response.setStatusCode()`
    + `response.bufferFactory().wrap(bytes)` + `response.writeWith(Mono.just(buffer))`。
    直接往 `response.getWriter` 写是 servlet API，会抛
  - `@ComponentScan("com.chuhezhe")` 已在 RagGatewayApplication 上，filter 包
    自动被扫到，不需要再手动 register
  - GlobalFilter 的 order 越小越早执行；放 -100 确保在路由匹配前判断 JWT，
    但要晚于 Spring Cloud Gateway 自身的几个前置 Filter（它们一般 order <
    -100000）
  - 完整 Feign 透传 + 网关拦截的活体校验要等 US-012 start-all.sh 把完整
    stack 起来后、US-013 e2e-smoke.sh 跑过再算数；当前 US-009 只保证编译
    和逻辑自洽
